# Tests

## Knowledge Cards Spaces And Checklists

### 1. Пространства

- `/cards` показывает три изолированных пространства:
  - `жизнь`;
  - `работа`;
  - `учёба`.
- `/cards/new` без `workspace` сначала показывает выбор пространства.
- Выбор `учёба` открывает форму карточки с `workspace=study`.
- API `/api/cards?workspace=study` возвращает только карточки учебного пространства.
- Legacy карточки без `workspace` продолжают нормализоваться в `life`.

### 2. Гибкое содержание

- Карточка может хранить обычное текстовое содержание.
- Карточка может хранить чек-лист с пунктами:
  - `id`;
  - `text`;
  - `checked`.
- Пустые пункты чек-листа отбрасываются при нормализации.
- В просмотре карточки чек-лист отображается как список checkbox-пунктов.
- Переключение checkbox сохраняет обновлённый статус через card update API.
- Поиск и preview используют текст пунктов чек-листа через общий fallback.

### 3. Автотесты

- `tests/integration/domain/cards-normalization.test.ts` покрывает `study` workspace и нормализацию чек-листов.
- `tests/integration/api/cards.route.test.ts` проверяет фильтрацию API по `workspace=study`.
- `tests/e2e/cards/cards-workspace-entry.spec.ts` проверяет, что на входе в базу знаний видны все три пространства.
- `tests/e2e/cards/cards-workspace-crud.spec.ts` проверяет создание учебной карточки с чек-листом.

## Sound Color Trainer MVP

### 1. Сохранение ассоциаций

- При первой калибровке ноты сохраняются:
  - `noteId`;
  - выбранный цвет;
  - `reactionTime`;
  - `confidenceLevel`.
- После сохранения у ноты появляется `baseColor`.
- После полной калибровки 12 нот статус пользователя перестаёт быть `new`.

### 2. Проверка совпадений

- В режиме `звук -> цвет` правильный выбор считается `correct = true`, если выбранный цвет совпадает с сохранённым `baseColor`.
- В режиме `звук -> цвет` неправильный выбор считается `correct = false`, если выбран другой цвет.
- В режиме `цвет -> нота` правильный ответ считается `correct = true`, если выбрана исходная нота.

### 3. Детекция нестабильности

- Нота попадает в список проблемных, если:
  - цвет меняется между калибровочными попытками;
  - стабильность ниже порога;
  - уверенность низкая;
  - средняя реакция слишком медленная.
- Минимальный вывод в статистике:
  - `noteId`;
  - причины проблемности;
  - текущая стабильность.

### 4. Минимальные ручные проверки UI

- Калибровка не показывает пользователю таблицу `нота -> цвет` заранее.
- Выбор цвета в калибровке работает только после проигрывания ноты.
- После клика по базовому цвету открывается popover уточнения оттенка.
- До нажатия `Подтвердить` выбор не считается ответом.
- `Esc` отменяет уточнение и ничего не сохраняет.
- Клик вне popover отменяет уточнение и ничего не сохраняет.
- По истечении окна реакции попытка не сохраняется автоматически.
- Данные восстанавливаются после перезагрузки страницы из локального хранилища.

### 5. Ручные проверки режимов закрепления

- После сохранения хотя бы одной ассоциации становится доступен запуск режимов упражнений.
- Выбор режима происходит на одном экране без перехода на отдельные страницы.
- В standalone prototype есть явные `stage` и `phase` selectors.
- Для `ASSIGNMENT` упражнений нет: есть только назначение `baseColor` для 12 нот одной октавы.
- Для `NOTES_IN_ONE_OCTAVE` режимы `fixation` не требуют угадывания из полной палитры.
- Для `NOTES_IN_ONE_OCTAVE` режимы `test` показывают только цвета текущего subset.
- Для `INTERVALS` на старте используются только ограниченные интервалы, а не весь набор сразу.
- Для `MELODY_IN_ONE_OCTAVE` режимы не используют octave-shift.
- Для `OCTAVES` `getColorForOctave()` действительно меняет только `lightness`, не `hue`.
- В `fixation` нет агрессивного результата `правильно/неправильно`.
- В `test` после завершения показывается простой результат.

### 6. Автотесты

- `tests/integration/domain/sound-color-trainer.test.ts` покрывает:
  - сохранение ассоциаций;
  - проверку совпадений;
  - совместимость со старым `HEX`-форматом;
  - близость оттенков внутри одной базовой группы;
  - детекцию нестабильности.
