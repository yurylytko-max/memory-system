# Coding Rules

## Core Principles

- Сохраняй приложение рабочим в production прежде всего.
- Предпочитай минимальные, локальные, понятные изменения вместо широких рефакторингов без необходимости.
- Если затрагиваешь продуктовую логику, проверяй не только UI, но и API/store слой.
- Если в коде есть временный обход, помечай его как технический долг в `docs/open-issues.md` или `docs/current-state.md`.

## Stack Rules

- Язык: TypeScript.
- Фреймворк: Next.js App Router.
- UI: React 19, shadcn/ui, Plate editor, Tailwind utilities и inline styles там, где это уже принято в модуле.
- Данные в серверной логике хранятся через `lib/server/*-store.ts`.

## File And Architecture Rules

- UI-страницы живут в `app/`.
- HTTP API живут в `app/api/`.
- Общая клиентская и domain-логика живёт в `lib/`.
- Серверный доступ к данным и внешним сервисам живёт в `lib/server/`.
- Переиспользуемые UI-компоненты живут в `components/`.
- Документация для агентов живёт только в `docs/`.

## Editing Rules

- Не возвращай удалённый `study 2.0`, если нет прямого запроса.
- Для учебников используй только `study-3` стек.
- Сохраняй совместимость маршрутов и редиректов, если пользователь отдельно не просит их поменять.
- Не удаляй auth-слой вокруг `/login`, cookie и middleware/proxy без явной задачи.
- Если редактируешь layout, proxy или deploy cleanup, учитывай разницу между Node runtime и Edge/runtime ограничениями.

## TypeScript And Build Rules

- Пиши типы явно там, где это повышает читаемость или защищает API-контракт.
- Не полагайся на `any`, если можно описать нормальный тип.
- Помни, что сейчас сборка допускает `typescript.ignoreBuildErrors = true`; не усиливай эту проблему новыми типовыми долгами.
- По возможности исправляй найденные типовые дефекты в затронутой области, даже если build их сейчас пропускает.

## React Rules

- Следуй существующему стилю файла; не устраивай косметический рефакторинг без пользы.
- Не добавляй `useMemo` и `useCallback` по привычке; только если это реально нужно.
- Для клиентских компонентов явно ставь `"use client"` только там, где это действительно требуется.
- Предпочитай простые пропсы и явные данные, а не скрытую магию.

## API Rules

- Любое изменение UI, которое меняет поведение данных, должно быть согласовано с соответствующим API route.
- Любое изменение API, которое меняет формат ответа или хранения, должно быть отражено в `docs/current-state.md`.
- Для защищённых endpoint'ов учитывай авторизацию через `proxy.ts` и `lib/auth.ts`.

## Persistence Rules

- Для новых сущностей стремись иметь парный слой:
  - `lib/<domain>.ts`
  - `lib/server/<domain>-store.ts`
- Не размазывай чтение/запись данных по UI-слою.
- Для migration-like изменений учитывай cleanup старых данных и документируй судьбу legacy-форматов.

## Study Rules

- Активная реализация учебников: `study-3`.
- Старые пути `/study` считаются legacy redirect surface.
- Любая новая функция учебников должна сразу добавляться в `study-3`, а не в отдельную параллельную ветку.
- Если трогаешь parsing, HTML pipeline, blob-файлы или Gemini integration, обнови `docs/current-state.md`.

## Planner Rules

- Дневные планы рассматриваются как производные от общих планов.
- Синхронизация статусов и удалений между дневными и исходными планами не должна ломаться.
- Удаление дневного плана и изменение задач внутри него нужно проверять как на уровне UI, так и на уровне route/store логики.

## Documentation Rules

- После значимых изменений обновляй `docs/`.
- Не держи changelog в голове; записывай факты в `docs/change-log.md`.
- Если решение временное, прямо называй его временным.
- Если есть неопределённость, фиксируй её в `docs/open-issues.md`, а не оставляй неявно.

## Verification Rules

- Минимум: релевантный `npm run lint`.
- При инфраструктурных и route-изменениях по возможности проверяй `npm run build`.
- Если локальный runtime нестабилен, фиксируй это честно и используй альтернативные проверки.
- Не заявляй о проверке, если она реально не была выполнена.

## Git Rules

- Избегай force-push без явного запроса.
- Если удалённая история ушла вперёд, сначала интегрируй её безопасно.
- После нетривиальных merge/push ситуаций обновляй `docs/current-state.md`, если это влияет на рабочий процесс команды.
